Heading Scale: Minimum 1.3× Step
Effective dashboards use at least a 1.3× size step between heading levels. H1 at 28px, H2 at 20px, body at 14px creates a clear scanning ladder. When all text is 14–16px, the eye has no anchor and must read everything to find what matters.

Body Font: 14–16px Minimum
Sub-14px body text forces users to zoom or squint. iOS Safari automatically inflates small text, which breaks your layout. Google's mobile-friendliness test flags it directly. Start at 16px for body, drop to 14px minimum for secondary text, 11px only for labels and captions.

Typography
One Typeface, Two Weights
The fastest way to make a UI look professional is to commit to one system font and two weights (400 for body, 600–700 for headings). Every additional typeface or weight decision is an opportunity for inconsistency. System fonts (SF Pro, Segoe UI, -apple-system) render crisply at every size and load instantly.

Data Labels: Uppercase + Letter Spacing
Metric labels should be visually subordinate to the number they describe. Uppercase + letter-spacing at 10–11px muted creates instant semantic separation: label = context, number = content. This pattern appears in Stripe, Baremetrics, and every well-designed analytics dashboard because it works at a glance.

Spacing
The 8-Point Grid System
Every spacing value should be a multiple of 8 (8, 16, 24, 32, 48, 64). This creates natural visual rhythm that makes layouts feel "right" even if users can't say why. Tailwind's default spacing scale, Material Design, and Apple HIG all use 4/8-based grids. Random spacing (13px here, 22px there) is the most common cause of layouts that look "almost professional."

Whitespace Ratio ≥ 20%
Layouts where content fills ≥80% of the screen feel overwhelming. A whitespace ratio of at least 20% — the fraction of the viewport with no content — signals professional restraint. Whitespace is not wasted space; it's the frame that makes content legible. Cramped UIs signal that design was an afterthought.

Button Consistency: One Primary Style
Every button variant is a visual choice that costs user attention. A primary (filled), secondary (outlined/ghost), and destructive (red) pattern covers 95% of use cases. Six different button shapes, sizes, and colors signal no design system — users can't build a mental model of what's clickable and what's important.

Empty states are high-value real estate — they appear when users are most likely to feel lost. A good empty state includes a visual cue (icon or illustration), a friendly title, a brief explanation, and a primary action. "No data found." tells the user nothing about what to do next. Shadow Scale: 2–3 Levels Maximum
Shadows communicate z-axis depth. A design system should define 2–3 shadow levels (flat → slight → elevated) and use them consistently. Arbitrary shadow values per card signal that each component was styled independently — shadows become visual noise rather than semantic information. Linear, Vercel, and Stripe all use a single subtle shadow for cards.

Accessibility
WCAG AA Contrast: 4.5:1 Minimum
WCAG 2.1 Level AA requires 4.5:1 contrast for normal text and 3:1 for large text (18px+). Low-contrast text looks stylish in mockups and unreadable in production — especially for users with vision impairments or on bright displays. Contrast failures are the most common WCAG issue and the easiest to fix: adjust lightness, not hue.

Accessibility
Never Use Color Alone for State
8% of men have red-green color blindness. Active/selected states communicated only via color change are invisible to them. Combine color with a shape cue: a border, an underline, a background change, or a bold weight. Navigation active states must be distinguishable by shape, not just color, per WCAG 1.4.1.

Accessibility
Focus Rings: Never Remove, Only Style
outline:none on focused elements removes all keyboard navigation cues. Keyboard users — including users who can't use a mouse, power users, and screen reader users — are left with no way to track where they are. WCAG 2.4.7 (Level AA) requires visible focus. Replace the default browser ring with a styled one: outline + offset in your accent color.

Single-Column Forms Complete Faster
Nielsen Norman Group studies show single-column forms are completed 15.4 seconds faster than multi-column equivalents. Eye-tracking data confirms users scan forms in a single vertical sweep — multi-column layouts interrupt that flow with horizontal jumps. Use multi-column only for tightly related short fields (City + State + ZIP).

Forms
Labels Above Fields, Never Placeholders-Only
Placeholder text disappears when the user starts typing — making it impossible to check what a field expects. This is especially harmful on long forms where users tab between fields or need to review entries. WCAG 3.3.2 requires visible labels. Use placeholder text only for examples or hints, never as a substitute for a label.

Mobile
Touch Targets: 44×44px Minimum
The average adult fingertip covers 44–57px. Targets smaller than 44px cause tap errors — users hit adjacent elements or miss entirely. Apple HIG mandates 44×44pt. WCAG 2.5.8 (Level AA) requires 24px with adequate spacing. Material Design recommends 48px for primary actions. Use padding to reach target size rather than increasing font size.

Viewport Meta Tag: Non-Negotiable
Without the viewport meta tag, mobile browsers render pages at ~980px and scale them down — text becomes a blur of 3px characters. It's the #1 mobile usability failure and disqualifies the page from Google's mobile-first index. It takes 30 seconds to fix and has zero downsides. If you remember only one mobile rule, make it this one.

Mobile
Fluid Layouts: Replace px Width with max-width
Fixed-pixel card widths are the root cause of most mobile layout breaks. A 600px card on a 375px phone clips, overlaps, or forces horizontal scroll. Fluid layouts — width: 100%; max-width: Xpx — work across all screen sizes without breakpoints. Use CSS Grid or Flexbox with flex-wrap: wrap to let cards reflow naturally on small screens.
